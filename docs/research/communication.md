# Communication and Collaboration

Effective collaboration depends on making information easy to find, understand, and share. The goal of ECHOLab's communication practices is not to prescribe a single communication style, but to ensure that important information remains accessible to current and future collaborators.

## Communication Channels

ECHOLab uses a variety of communication tools for different purposes:

| Tool     | Typical Use                                                   |
| -------- | ------------------------------------------------------------- |
| Slack    | Day-to-day communication and project discussions              |
| Email    | Communication outside the lab and formal correspondence       |
| GitHub   | Code review, project documentation, and technical discussions |
| Notion   | Project planning, meeting notes, and task tracking            |
| Overleaf | Manuscript development and collaboration                      |

Slack is generally the preferred platform for routine communication within the lab.

## Public by Default

Direct messages are appropriate for quick questions, scheduling, and other minor communications.

However, substantive project discussions, decisions, and information that may be useful to future collaborators should generally occur in project channels or be documented elsewhere.

Future collaborators should be able to understand the history of a project without relying on access to private conversations.

## Project Channels

Active projects are encouraged to maintain dedicated Slack channels when practical.

Project channels provide a shared location for:

* Project discussions
* Announcements and updates
* Questions and troubleshooting
* Sharing resources and references
* Coordination among collaborators

When possible, project communication resources should be linked from the project's GitHub repository.

## Document Important Decisions

Important project decisions should be documented in a location that collaborators can easily discover and reference.

Examples may include:

* Project README files
* Notion pages
* Meeting notes
* GitHub issues or pull requests
* Project Slack channels

The specific location is less important than ensuring that information remains accessible to collaborators.

## Reduce Single Points of Failure

Information that is important to a project should not exist solely in one person's memory, email inbox, or direct messages.

Whenever practical, project knowledge should be documented in locations that remain available to collaborators over time.

Doing so reduces onboarding costs, improves continuity as researchers join and leave the lab, and helps ensure that projects remain understandable and reproducible.

## Meeting Notes

Meeting notes are encouraged for recurring project meetings and collaborative workstreams.

Notes can serve as a useful record of:

* Decisions made
* Open questions
* Action items
* Relevant resources and references

Meeting notes should be stored in a location accessible to project collaborators whenever practical.

## Guiding Principle

> Make important information discoverable.

The objective is not to document everything. Rather, it is to ensure that information that is important to a project's success can be located, understood, and used by future collaborators.
